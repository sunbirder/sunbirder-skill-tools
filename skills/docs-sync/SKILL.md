# 文档同步

项目持续开发过程中，代码在变，文档容易过时。这个技能负责让文档重新对齐代码。

## 流程

### 第一步：确定范围

问用户要同步的范围：

- 全部文档都要检查？还是指定某个文件/目录？
- 只检查近期变更相关的文档？还是全面对比？

通常先 `git log --oneline -20` 了解最近的改动，以此为线索缩小范围。

### 第二步：扫描代码

针对确定的范围，读取对应的代码文件：

- 文件结构是否变化？（新增/删除/重命名）
- 函数、类、接口是否变化？（签名、行为）
- CLI 命令、配置项是否变化？
- 安装/启动方式是否变化？

### 第三步：对比文档

逐个读取范围内的文档，逐项对照代码：

- **过时内容** — 文档描述的行为和代码不一致
- **遗漏内容** — 代码有但文档没提到的功能
- **已删除内容** — 文档写了但代码里不存在的

用表格列出所有发现：

| 文档文件 | 问题 | 类型 | 严重程度 |
|----------|------|------|----------|
| docs/guide/xxx.md | 函数签名已改为... | 过时 | 高 |
| docs/guide/yyy.md | 未记录新增的 --flag | 遗漏 | 中 |

### 第四步：确认后修复

向用户展示发现的问题列表，逐条确认后修改文档。修改原则：

- 只改和代码不一致的部分，不顺便改写风格
- 每条修改要有代码依据（文件路径 + 行号）
- 保持原文档的结构和行文风格
- 如果文档提到 `git log` 可验证的操作，确保结果匹配

## 原则

- **以代码为准** — 文档服务于代码，不对的地方以代码事实为准
- **最小改动** — 只修问题，不重写
- **有据可查** — 每条修改标注代码出处
- **先确认再改** — 列完问题清单让用户 review，不改不该改的
